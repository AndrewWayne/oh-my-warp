## 2. What's not done

### 2.1 Gap 1 Part C — auto-share-on-Phone-click wiring

**Symptom.** When the user clicks Phone, the daemon starts and the pair URL is auto-copied to the clipboard, but **omw-remote spawns a sibling shell** when the phone connects via `/ws/v1/pty/:id` — not the active Warp pane. The original Gap 1 symptom remains user-visible.

**TODO marker.** [`vendor/warp-stripped/app/src/terminal/view/use_agent_footer/mod.rs:304-312`](../vendor/warp-stripped/app/src/terminal/view/use_agent_footer/mod.rs).

**Why it didn't land.** The click handler is `TerminalView::handle_use_agent_footer_event`. `TerminalView` is intentionally manager-agnostic (`vendor/warp-stripped/app/src/terminal/writeable_pty/terminal_manager_util.rs:20` says so) — it has no field holding the `TerminalManager`. The concrete `local_tty::TerminalManager` (the only impl exposing `event_loop_tx()` / `pty_reads_tx()` accessors from Gap 1B) lives behind the parent `PaneView`'s `child_data` slot (`vendor/warp-stripped/app/src/pane_group/pane/terminal_pane.rs:187`). Reaching it requires:

```
self.pane_stack.upgrade()
  → find owning PaneView
  → child_data()
  → downcast Box<dyn TerminalManager> via as_any()
```

That cuts across the manager-agnostic boundary and isn't a surgical change.

**Recommended fix path.** Two options, ordered by invasiveness:

**A. Push the share down through an action variant.** In a callsite that has structural access to both the pane and its terminal manager (the parent that creates `TerminalView`), pass the cloned `event_loop_tx` and `pty_reads_tx` into `AgentInputFooterAction::ToggleOmwPair`. The `TerminalView` handler then has them in scope.

  Sketch:
  ```rust
  // wherever AgentInputFooterAction::ToggleOmwPair is dispatched:
  let manager = pane_view.child_data().as_any().downcast_ref::<LocalTerminalManager>()?;
  let event_loop_tx = manager.event_loop_tx();
  let pty_reads_tx = manager.pty_reads_tx();
  AgentInputFooterAction::ToggleOmwPair {
      event_loop_tx,
      pty_reads_tx,
      pane_name: pane_view.title().to_string(),
  }
  ```

  Then in the handler:
  ```rust
  // after state.start():
  if let Some(registry) = state.pty_registry() {
      let handle_fut = crate::omw::pane_share::share_pane(
          pane_name, event_loop_tx, pty_reads_tx, registry,
      );
      let runtime = state.runtime_handle().expect("runtime up after start");
      runtime.block_on(async move {
          if let Ok(handle) = handle_fut.await {
              state.push_share_handle(handle);
          }
      });
  }
  ```

  Add a `pub fn push_share_handle(&self, handle: PaneShareHandle)` and a `Vec<PaneShareHandle>` field on `OmwRemoteState`. Drop them on `stop()`.

**B. Pane-registry on `OmwRemoteState`.** When a `local_tty::TerminalManager` is constructed, register itself with `OmwRemoteState`'s pane registry. On `start()`, walk the registry and share each pane (or just the most recent / focused one). More invasive — touches manager construction — but gives "share all panes automatically" out of the box.

  Don't do this without an RFC. A only.

**Estimated effort.** ~1-2 hours including testing.

**Acceptance once landed.** `docs/v0.4-thin-functional-gaps.md` Gap 1 acceptance #3: open warp-oss, run a long-lived command in the active pane (`tail -f some.log` on Unix; `Get-Content -Wait some.log` on Windows), click Phone, scan QR with phone, observe the live tail on phone within 1 second of pairing.

### 2.2 Gap 2 — reactive `View<>`-backed dialog

**Symptom.** Click Phone → URL is auto-copied to clipboard + printed to stderr in a formatted block. **No visible in-app dialog.** A user double-clicking `warp-oss.exe` from Explorer never sees the URL unless they paste from the clipboard or read stderr from the launching console.

**Module that's ready to plug in.** [`vendor/warp-stripped/app/src/omw/pair_modal.rs`](../vendor/warp-stripped/app/src/omw/pair_modal.rs) — pure-text formatter (`format_pair_modal_text`, `format_pair_modal_text_block`), 4 unit tests. [`vendor/warp-stripped/app/src/omw/qr.rs`](../vendor/warp-stripped/app/src/omw/qr.rs) — `render_pair_url_svg` + `render_pair_url_grid`, 2 unit tests.

**Why it didn't land.** Warp's `Dialog` widget at `vendor/warp-stripped/app/src/ui_components/dialog.rs` integrates via `ViewHandle<MyDialog>` on `WorkspaceView`. The integration path requires ~7 new sites in `workspace/view.rs` (~22K lines): `ViewHandle` field, build, focus, render conditional, action enum, event handler, state flag. Not surgical.

**Recommended fix path.**

1. **Read the settings dialog** as the closest pattern — find its `View` impl and the `WorkspaceView` integration sites (search `vendor/warp-stripped/app/src/workspace/view.rs` for `settings_dialog` or `SettingsDialog`).

2. **New `View` impl** at `vendor/warp-stripped/app/src/omw/pair_modal.rs` (extend the existing module). The `View::render` consumes `format_pair_modal_text(content)` row-by-row and adds:
   - A QR canvas that renders `qr::render_pair_url_grid(pair_url)` as native rectangles (one rect per `true` cell). Or: `qr::render_pair_url_svg` and a generic SVG renderer if Warp has one — search for `Svg` in `ui_components/`.
   - `[Copy]` button → calls `ctx.clipboard().write(ClipboardContent::plain_text(pair_url))`.
   - `[Show QR]` toggle → flips a `bool` in the modal state, conditional-renders the QR canvas.
   - `[Stop Daemon]` button → calls `OmwRemoteState::shared().stop()` then closes the dialog.
   - `[Close]` button → closes the dialog without stopping.

3. **Subscribe to status_rx + tailscale poll.** The modal subscribes to `OmwRemoteState::shared().status_rx()` (Gap 3 watch channel) so the status line + pair URL refresh on every transition. Tailscale status doesn't change at runtime within a session, so a single `detect_status()` call at modal-open is fine.

4. **Wire into `WorkspaceView`.** Add a `pair_modal: Option<ViewHandle<PairModalView>>` field. The `ToggleOmwPair` action handler at `terminal/view/use_agent_footer/mod.rs:294` builds the view and assigns it. Replace the current `eprintln!` + clipboard-only fallback with the dialog open call.

5. **Remove the `eprintln!` toast** once the dialog is reliably visible. Keep clipboard auto-copy on open as belt-and-suspenders.

**Estimated effort.** ~2-3 hours. The biggest cost is reading enough of `workspace/view.rs` to find the right insertion sites. Pair this work with someone who knows Warp's view framework idioms.

**Acceptance once landed.** `docs/v0.4-thin-functional-gaps.md` Gap 2 acceptance #1-#3, #5, #6: modal opens within 200ms with pair URL displayed; "Show QR" reveals scannable QR; "Copy" places URL on clipboard; "Stop Daemon" terminates daemon and updates modal status; no stderr URL prints required.

### 2.3 Gap 2 — `Paired devices: N` live count

**Symptom.** The pair-modal text formatter renders `Paired devices: ?` because no source bumps the count. Documented in [`vendor/warp-stripped/app/src/omw/pair_modal.rs:41-44`](../vendor/warp-stripped/app/src/omw/pair_modal.rs).

**Recommended fix path.**

1. **Daemon-side hook.** Add a callback parameter to `omw_remote::ServerConfig`:
   ```rust

   ```
   `omw-remote`'s `Pairings::redeem` handler in `crates/omw-remote/src/handlers/pair.rs` (or wherever it lives — search `redeem`) invokes this callback after a successful redeem.

In `bring_up_daemon`, set `config.on_pair_redeem` to a closure that loads the count from the `pairings` table (or just bumps the watch by 1 per redeem) and calls `paired_count_tx.send_replace(count)`.

3. **Modal subscribes** alongside `status_rx`. Replace `paired_device_count: None` at the build site (`use_agent_footer/mod.rs:325`) with `Some(*paired_count_rx.borrow())`.

**Estimated effort.** ~30-45 min. Simpler than the dialog itself.

## 3. Known process problems / pitfalls

### 3.1 codex on Windows — `ShellExecuteExW failed: 1223` sandbox helper

**Symptom.** When piping a prompt + diff into `codex exec --skip-git-repo-check --sandbox read-only -`, codex's internal helper that spawns PowerShell for file reads / git commands fails with:

```
windows sandbox: orchestrator_helper_launch_canceled:
ShellExecuteExW failed to launch setup helper: 1223
```

Error 1223 = `ERROR_CANCELLED` (operation cancelled by user — i.e. UAC denied). This happened reproducibly on this Windows host on every codex invocation that tried to grep / read files.

**Workaround used.** Codex still rendered a verdict from the inlined diff alone (the prompt streamed the full diff via heredoc, so codex didn't strictly need filesystem access). But its verdicts were "static review only" — it explicitly noted it could not run tests or verify cross-file consistency.

**For the next session.** Either:
- (a) Run codex on Linux/macOS instead, or under WSL on this same machine.
- (b) Disable the read-only sandbox: `--no-sandbox` (security trade-off — review the codex docs).
- (c) Run codex in a separate elevated PowerShell session where UAC will pre-approve the helper.
- (d) Continue using a `superpowers:code-reviewer` Agent as the primary reviewer; treat codex as best-effort second eye when its sandbox lets it run.

This session went with (d). It worked but added latency on each cross-review.

### 3.2 Dead-code warnings (5 currently surfaced)

`cargo build -p warp --bin warp-oss --features omw_local` emits 5 warnings:

```
warning: function `render_pair_url_svg` is never used         [omw/qr.rs]
warning: function `render_pair_url_grid` is never used         [omw/qr.rs]
warning: methods `toggle` and `stop` are never used            [omw/remote_state.rs]
warning: field `tailnet` is never read                          [omw/tailscale.rs]
warning: function `unserve` is never used                       [omw/tailscale.rs]
```

**Status of each:**

- `render_pair_url_svg` / `render_pair_url_grid` — **dormant until Gap 2 reactive dialog lands**. The dialog renderer will consume these. Leave as-is.
- `toggle` / `stop` — `OmwRemoteState::toggle` and `OmwRemoteState::stop` were the original Wiring-5 button entry points. After Gap 3, the click handler calls `start()` directly (Gap 2 toast surface) — toggle isn't called. **Needs revisit when Gap 2 dialog lands**: the dialog's "Stop Daemon" button should call `stop()`, restoring the warning.
- `tailnet` field — populated by `detect_status` but `format_pair_modal_text` derives the same FQDN from `local_hostname` (which already contains the tailnet suffix). **Either remove the field or document why it's kept.** Recommend removing in the Gap 2 polish PR; if some callsite later wants the bare tailnet domain, revive it then.
- `unserve` — `OmwRemoteState::stop` calls it, but `stop()` itself is dead per above. Cascading dead-code. Revives when Gap 2 dialog wires "Stop Daemon".

**Don't `#[allow(dead_code)]` these blindly** — the warnings are accurate signals that Gap 2 wiring is incomplete. Let them ride until the dialog lands; cleanup is then trivial.

### 3.3 Gap 1B intentional `#[allow(dead_code)]`

[`vendor/warp-stripped/app/src/omw/pane_share.rs:30`](../vendor/warp-stripped/app/src/omw/pane_share.rs) module-level `#![allow(dead_code)]` is intentional and correct: the public API is unused until Gap 1 Part C wires it. **Remove this attribute** as part of the Part C PR; the unused-API warnings should clear naturally once `share_pane` is called.

### 3.4 Tokio runtime for `share_pane`

`share_pane` is `async` and spawns pumps via `tokio::spawn` (`pane_share.rs:95, 114, 151`). It must be called from inside a tokio runtime context. **The recommended call pattern** is:

```rust
let runtime = OmwRemoteState::shared().runtime_handle()?; // Gap 3 already exposes runtime_handle internally
let handle = runtime.block_on(async move {
    share_pane(pane_name, event_loop_tx, pty_reads_tx, registry).await
})?;
```

A naked `share_pane(...).await` from a non-runtime UI thread will panic. The Gap 1 Part C wiring must use `runtime_handle.block_on` or `.spawn`.

### 3.5 Hardcoded port `8787`

The bind port appears in three places in `vendor/warp-stripped/app/src/omw/remote_state.rs`:
- `DEFAULT_BIND` at line 26 (`"127.0.0.1:8787"`)
- `tailscale::serve_https(8787)` at line ~343
- `tailscale::unserve(8787)` at line ~215

If `DEFAULT_BIND` ever changes, the other two drift silently. **Suggest a `const PORT: u16 = 8787` next polish pass** and parse `DEFAULT_BIND` from `SocketAddr` in one place.

### 3.6 TRD overhead vs. vendor-code edits

TRD-formal (Overseer + Executor + dual cross-review at every stage gate) is a poor fit for `vendor/warp-stripped/` work. The framework is large, the test surface is small (most additions are integration-y), and the cost-per-change of running TRD ceremony is multiples of the change itself. This session went TRD-formal on Gap 1 part A (the registry refactor — where TRD's anti-cheating discipline mattered) and TRD-lite (single executor + single cross-reviewer) on Gaps 1B / 1C / 4 / 3 / 2 (vendor-side wiring). **For the continuation work**: TRD-lite is sufficient for the remaining Gap 1C and Gap 2 dialog wiring. Reserve TRD-formal for changes inside `crates/omw-*` MIT crates with clear unit-test surface.

## 4. Demo state today vs. target

### Today (after this session)

```
1. User runs warp-oss.exe.
2. User clicks Phone button.
3. Daemon starts on tailnet (127.0.0.1:8787 + Tailscale Serve auto-bootstrapped if installed).
4. Pair URL auto-copied to clipboard. Modal text printed to stderr (visible only if launched from a console).
5. Tooltip transitions: Stopped → Starting → "Stop phone pairing" within ~200ms (Gap 3 watch channel).
6. User pastes URL into any QR code generator (online).
7. User scans QR with phone.
8. Phone connects → omw-remote spawns A SIBLING SHELL (cmd.exe / /bin/sh).
9. Phone displays the sibling shell — NOT the Warp pane the user was looking at.
```

### Target (after Gaps 1C + 2 land)

```
1. User runs warp-oss.exe.
2. User opens a terminal pane and runs `tail -f some.log`.
3. User clicks Phone button.
4. Daemon starts; in-app modal opens within 200ms showing:
     Status: Running on https://laptop.tailnet.ts.net
     [QR canvas]
     Pair URL: https://laptop.tailnet.ts.net/pair?t=...
     [Copy] [Show QR] [Stop Daemon]
5. User scans QR with phone.
6. Phone connects → sees the LIVE TAIL of the Warp pane in real time.
7. User types on phone → laptop pane echoes the typed character.
8. Modal updates: Paired devices: 1.
9. User clicks [Stop Daemon] → modal updates to Stopped.
```

The **gap to target** = §2.1 + §2.2 + §2.3 above.
