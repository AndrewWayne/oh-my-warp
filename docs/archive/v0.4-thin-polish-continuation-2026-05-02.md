## 2. Decisions that override the 2026-05-01 plan

### 2.1 Tailscale Serve is no longer on the demo path

The 2026-05-01 doc treated Tailscale Serve as the "right" HTTPS path (Gap 4). This session ran into reality:

- Tailscale 1.96.x removed the legacy `tailscale serve --bg --https=PORT <target>` syntax. Modern usage is `tailscale serve --bg <port>` only — port 443 is implied via the tailnet hostname.
- Serve isn't enabled by default on a tailnet — the user has to visit `https://login.tailscale.com/f/serve` and click an admin-console toggle. Extra friction we don't want for a shipping product.
- `tailscale serve --bg <port>` can hang (cert provisioning) on first use, freezing the UI before the timeout patch landed. Even with the timeout, an 8 s wait on every Phone-click is bad UX.

Decision: **drop the Serve path entirely**. The daemon binds `0.0.0.0:8787` and the pair URL uses the tailnet IPv4 (`http://100.x.x.x:8787/pair?t=...`). The phone reaches us directly over the tailnet without any cert / Serve / enrollment.

Cost paid: the URL is plain HTTP, which means the **phone's browser** can't use `crypto.subtle` (WebCrypto Ed25519 keygen + the SHA-512 it depends on) or `getUserMedia` (in-app camera scan). Browsers gate both behind a "secure context" — HTTPS, localhost, or `file://`. Plain HTTP over a tailnet IP doesn't qualify.

Mitigation = §2.2.

### 2.2 Web Controller crypto is now pure-JS, no WebCrypto.subtle

To make the demo work over plain HTTP, `apps/web-controller/src/lib/crypto/ed25519.ts` no longer relies on `crypto.subtle` at all:

- The Ed25519 ops were already noble-ed25519 (pure JS) but noble-ed25519 v2 internally calls `crypto.subtle.digest('SHA-512', ...)` for hashing. That's the actual blocker.
- We pulled in `@noble/hashes` (~5 KB gzipped) and wired its `sha512` into `nobleEd.etc.sha512Async` / `etc.sha512Sync`. Hashing is now pure JS.
- Keygen, sign, verify all work over plain HTTP from any phone browser.

Implication for future hardening: if you eventually want HTTPS (Tailscale Serve, self-signed cert, or `tailscale cert`), nothing here gets in the way — noble-ed25519 stays the same, and the secure-context path "just becomes available" again. But it's no longer required for v0.4-thin.

### 2.3 Camera scan is hidden in non-secure contexts

`Pair.tsx` now does a `cameraScanAvailable()` check before rendering the "Start scan" button. The check covers:
- `navigator.mediaDevices` is defined
- `navigator.mediaDevices.getUserMedia` is a function
- `window.isSecureContext === true`

When false (the plain-HTTP-over-tailnet case), the button is replaced with a one-line hint: "Open the host's pair URL directly (the page auto-pairs from `?t=`), or paste the URL below." No more cryptic "couldn't start camera: undefined is not an object."

The primary pairing flow is **phone OS camera scans QR → opens URL in phone browser → page auto-fires redeem**, so the in-app camera scan was always a fallback. Hiding it on insecure contexts is graceful degradation, not a feature loss.

### 2.4 Pair page now auto-redeems on mount

When the phone arrives at `http://<host>:8787/pair?t=<token>`, the page now:

1. Extracts `t` from BOTH the path (`/pair/:t`) and the query string (`?t=<token>`). Previously only the path-form auto-prefilled.
2. Auto-fires `redeemPairing` once — guarded by a `useRef` so re-renders don't loop. Default device name is the user-agent-derived guess.
3. On success, `savePairing` to IndexedDB and `navigate("/")`.

User opens URL → "just paired." No tapping needed.

## 3. New deferrals + known issues introduced this session

### 3.1 `0.0.0.0` bind exposes the daemon to the LAN, not just the tailnet

The bind change in `66a6bfd` makes the daemon reachable on every interface — tailnet, LAN, and any public IP the host has. WS Origin pinning still rejects upgrades whose Origin isn't in `pinned_origins` (`[loopback, http://<tailnet-ip>:8787]`), but:

- HTTP routes that aren't WS upgrades — the embedded static assets at `/` and the `POST /api/v1/pair/redeem` endpoint — DON'T do Origin pinning. They rely on per-route auth (signed pair token / capability token).
- The pair-redeem endpoint validates the one-time token + 10 min TTL + nonce. Brute force isn't viable on an 80-bit token in 10 min. So this is acceptable, but worth noting as a defense-in-depth gap.

**Continuation work**: bind to a specific tailnet interface IP instead of `0.0.0.0` when Tailscale is up. The IPv4 is already in `TailscaleStatus.tailnet_ipv4`. Just `bind = (tailnet_ipv4, 8787).parse()` instead of `0.0.0.0:8787`. Falls back to `0.0.0.0` (or to loopback-only) when Tailscale is missing.

Effort: ~30 min. Not blocking the demo.

### 3.9 (RESOLVED in `af2dba0` + regression test `eb09b90`) Cross-language canonical-bytes drift

**The biggest "everything green but nothing works" bug of the session.**

Symptom: phone-side `POST /api/v1/sessions` returned
`{"error":"signature_invalid"}` after pair-redeem succeeded.

Root cause: a one-byte difference between client and server canonical
encodings:

- **Server** (`crates/omw-remote/src/auth.rs::CanonicalRequest::to_bytes`):
  ```rust
  format!("{}\n{}\n{}\n{}\n{}\n{}\n{}\n{}\n", ...)
  ```
  Yields `METHOD\n...\nVERSION\n` — note the **trailing** `\n`.

- **Client** (`apps/web-controller/src/lib/crypto/canonical.ts`):
  ```ts
  lines.join("\n")
  ```
  Yielded `METHOD\n...\nVERSION` — **no** trailing `\n`.

Different bytes → different SHA-512 outputs → different Ed25519
signatures → server rejects with `signature_invalid`. The diff is
exactly one byte.

Why latent through every prior test:
- Rust e2e tests (`e2e_full_flow.rs`, `ws_via_registry.rs`,
  `http_pair_redeem.rs`) sign + verify in Rust, both sides use the
  trailing-\n format → match.
- JS unit tests (`vitest`) sign + verify in JS, both sides use the
  no-trailing-\n format → match.
- The `canonical.test.ts` actively pinned the bug:
  `expect(text.endsWith("\n")).toBe(false)`.
- No JS-client-vs-Rust-server test existed.

Fix in `af2dba0`: `lines.join("\n") + "\n"` in canonical.ts; updated
the canonical.test.ts assertion.

Regression guard in `eb09b90`: new Rust integration test
`crates/omw-remote/tests/js_client_canonical.rs` with two cases:
1. Builds canonical bytes the JS way, asserts byte-equality with
   `CanonicalRequest::to_bytes` for the same input.
2. Mints a host key + capability token, signs the JS-shaped canonical
   bytes with a device key, runs the same `Verifier::verify` path the
   live HTTP handler uses. Asserts verification succeeds.

If the JS encoding ever drifts again, test 1 surfaces the diff
immediately. If signature verification ever fails for legitimate
clients, test 2 catches it.

**Lesson for the next session:** unit-test green is necessary but not
sufficient. The next equivalent gap will be found by running an actual
phone (or curl + ed25519-dalek script) against the live daemon on every
spec-touching change.

### 3.10 (RESOLVED in `5fdcb2a` + regression test) WS frame ts format drift

Sister bug to §3.9. After fixing the HTTP canonical encoding, the phone
got past pair + session-create and into the Terminal page — but every
keystroke on xterm.js triggered an immediate WS close: `4401
auth_failed`. The server was running `verify_frame` on each inbound
frame and rejecting the device signature.

Same failure shape, different field:
- **Server** (`crates/omw-remote/src/ws/frame.rs::Frame::canonical_bytes`):
  used `self.ts.to_rfc3339()` (chrono default), which produces
  `2026-05-02T01:00:00.123+00:00` — the `+00:00` suffix.
- **Client** (`apps/web-controller/src/lib/pty-ws.ts::frameCanonicalBytes`):
  used the wire `ts` string verbatim. JS sent ts via `new
  Date().toISOString()` which always produces `2026-05-02T01:00:00.123Z`
  — the `Z` suffix.

Different bytes on the `ts` line of the canonical JSON → different
SHA-512 inputs → frame signature_invalid → Outbound::Close(4401,
"auth_failed").

Fix: new `format_ts()` helper in `frame.rs` uses
`to_rfc3339_opts(SecondsFormat::Millis, true)` — produces the `Z`
form, with fixed 3-digit millisecond precision matching JS. Both
`canonical_bytes` and `to_json` route through it so the wire `ts`
and the signed canonical bytes stay byte-identical. JS canonicalizes
inbound frames via the wire `ts` string verbatim, so output-frame
verification on the JS side also stays consistent.

Regression test: `crates/omw-remote/tests/js_frame_canonical.rs` —
mirrors `js_client_canonical.rs` for the WS path. Probe assertion
proved the `Z`-vs-`+00:00` diff before the fix; positive assertion
verifies a JS-shaped signed frame via `Frame::verify` after.

Both `js_client_canonical.rs` (HTTP signed-request) and
`js_frame_canonical.rs` (WS frames) now form a cross-language
encoding guard: any future drift in either canonical format breaks
one of these tests immediately, before users hit `signature_invalid`
on a phone.

**Lesson reinforced from §3.9:** there's a third place the JS+Rust
encodings could still drift and we haven't tested — the WS connect-
token bundle. It's signed by the same `apps/web-controller/src/lib/
pty-ws.ts::buildConnectToken` path, so it shares the HTTP canonical
bytes (which we fixed in §3.9) — but if any future signed bundle
appears (e.g. for revoke-pairing, audit-append), it needs its own
cross-language sign+verify regression test BEFORE the JS code
ships.

All four debug lines fire. Server-side: origin pinning passed, CT

3. **The 101 packet is being mangled in transit.** Tailscale shouldn't

### 3.11b (related, separate fix) Stale session UUID after daemon restart → WS 404

While debugging §3.11, an earlier log showed:

```
[omw-debug] ws_handler -> 404 session not in registry: d81968fa-...
```

Cause: `omw-server::SessionRegistry` is in-memory only. Daemon restart
wipes it. The phone's React Router URL `/terminal/warp-host/<UUID>`
retains the old UUID; retry hits the new daemon's empty registry and
404s.

This is independent of §3.11 — it surfaces only when the user restarts
the daemon while a phone holds an active terminal page. But it's a
realistic UX failure mode (the user kills warp-oss to test a new
build, expects the phone to reconnect on Retry).

**Fix (~15 min, in `apps/web-controller/src/pages/Terminal.tsx`):**
auto-retry with a fresh session when the URL sessionId doesn't work.

```ts
let actualSessionId = sessionId!;
let connection: PtyConnection | null = null;
try {
  connection = await connectPty({ pairing, sessionId: actualSessionId });
} catch (firstErr) {
  try {
    actualSessionId = await createDefaultSession(pairing, "main");
    connection = await connectPty({ pairing, sessionId: actualSessionId });
    // Replace URL so subsequent reloads don't retry the dead UUID.
    navigate(
      `/terminal/${encodeURIComponent(hostId!)}/${encodeURIComponent(actualSessionId)}`,
      { replace: true },
    );
  } catch (secondErr) {
    setErrorMsg(`Failed to connect: ${errStr(secondErr)}`);
    setStatus("error");
    return;
  }
}
```

Symptom on phone (last log captured):

```
[omw-debug] ws_handler entry: session_id=d81968fa-... origin=tailnet host=tailnet
[omw-debug] ws_handler -> auth ok device_id=9af5d704bab6ba68
[omw-debug] ws_handler -> 404 session not in registry: d81968fa-...
```

Cause: `omw-server::SessionRegistry` is in-memory only — every daemon
restart wipes it. The phone's React Router URL after pair was
`/terminal/warp-host/<UUID>`, where `<UUID>` was the session created
during the post-pair flow. After warp-oss was killed + rebuilt + relaunched
(several times tonight), the phone retained the old URL with the old UUID
in IndexedDB / browser state. The retry click reuses the stale URL → WS
upgrade hits a UUID that doesn't exist in the new daemon's registry →
404 `session_not_found`.

Browser-side, the WS upgrade gets a non-101 response → fires close event
1006-ish → `waitOpen()` rejects → Terminal.tsx setStatus("error") +
errorMsg. So this should actually surface as `error` on the phone, with a
Retry button. The "still connecting" reports earlier may have been a
mix of (a) this stale-UUID failure, and (b) the React close-handler
race in Terminal.tsx not catching the close fast enough.

**Fix (~15 min, in `apps/web-controller/src/pages/Terminal.tsx`):**

```ts
// Try the URL sessionId first (good for live-session re-attach after
// reload). If WS fails (session_not_found, daemon-restart, expired),
// fall back to creating a fresh session.
let actualSessionId = sessionId!;
let connection: PtyConnection | null = null;
try {
  connection = await connectPty({ pairing, sessionId: actualSessionId });
} catch (firstErr) {
  try {
    actualSessionId = await createDefaultSession(pairing, "main");
    connection = await connectPty({ pairing, sessionId: actualSessionId });
    // Optional: replace URL so refresh uses the live UUID.
    navigate(
      `/terminal/${encodeURIComponent(hostId!)}/${encodeURIComponent(actualSessionId)}`,
      { replace: true },
    );
  } catch (secondErr) {
    setErrorMsg(`Failed to connect: ${errStr(secondErr)}`);
    setStatus("error");
    return;
  }
}
```

Acceptance:
- Pair phone, navigate to /terminal/.../UUID. xterm opens, prompt
  appears, typing works.
- Kill daemon. Restart. On phone, hit Retry. Terminal page automatically
  re-creates a session, connects, prompt appears. URL bar updates to
  the new UUID (via `navigate({ replace: true })`).

This is the ONLY change needed to clear tonight's last observed
failure. Everything upstream (origin pinning, signed CT, canonical
bytes, ts skew, frame format) is now demonstrably working — the WS
upgrade succeeded all the way through `auth ok` before failing on
the registry lookup.

### 3.12 Pair token is frozen for the daemon's lifetime — does NOT auto-rotate

**This is a correctness gap, not a polish item.** It was missed during the
2026-05-01 session and not added to the original gap analysis.

`bring_up_daemon` issues exactly one pair token at daemon startup
(`vendor/warp-stripped/app/src/omw/remote_state.rs::bring_up_daemon`,
~line 416: `let token = pairings.issue(PAIR_TTL)?`). That token is embedded
in `OmwRemoteStatus::Running.pair_url` and the URL never changes for the
lifetime of the daemon process. Consequences:

- A user who pairs phone A successfully cannot pair phone B from the same
  daemon session — phone B sees `token_already_used` (single-use token).
- After 10 minutes (`PAIR_TTL`), the token expires; subsequent pair attempts
  see `token_expired`. The user has to stop+start the daemon to get a fresh
  token.
- The pair-browser HTML page (`pair_browser.rs`) is written once at
  Phone-click and persists at `<temp>/omw-pair.html` with the stale URL
  even after the token is gone.

Even worse, the `Arc<Pairings>` is consumed by `omw_remote::serve(config)`
when the daemon starts — the click handler has no reference to it after
that, so it can't issue fresh tokens against the same database without
opening its own SQLite connection.

**Fix path (straightforward, ~30-45 min):**

1. Mirror the `pty_registry` pattern from Gap 1B. Clone the `Arc<Pairings>`
   before moving it into `ServerConfig`:

   ```rust
   let pairings_for_state = pairings.clone();
   let config = ServerConfig { pairings: Some(pairings), ... };
   // return pairings_for_state alongside the existing init tuple
   ```

2. Store it in `OmwRemoteState::Inner.pairings: Option<Arc<Pairings>>`.

3. Add `pub fn issue_pair_token(&self) -> Result<String, String>` on
   `OmwRemoteState`. Returns a fresh `?t=<token>` URL using the current
   `pair_origin`.

4. In `use_agent_footer/mod.rs::ToggleOmwPair`, every Phone click after
   the daemon is up calls `issue_pair_token()` and rebuilds the
   pair-browser HTML with the fresh token. Don't hold a stale URL in
   `OmwRemoteStatus::Running` — either:
   (a) drop `pair_url` from the status struct and have the UI compute it
       on demand, or
   (b) keep a cached "last issued" URL but always re-issue on user-visible
       events (modal open, click).

5. (Optional) Periodic auto-refresh: a tokio interval that re-issues every
   ~5 min so a stale browser tab can refresh and pick up a live token via
   a `GET /api/v1/pair/current` endpoint. Out of scope for v0.4-thin.

**Acceptance:** click Phone → pair phone A successfully → click Phone again
→ pair phone B successfully (different token, no `token_already_used`). And:
click Phone → wait 11 minutes → click Phone → fresh token issued (not
expired).
