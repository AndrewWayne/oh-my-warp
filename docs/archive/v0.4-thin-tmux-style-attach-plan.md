## 1. Why we're doing this

Stage C of the multi-session design tried to surface "see history when I
reconnect" by storing a bounded byte ring in `omw-server::SessionRegistry`
and replaying it on every WS subscribe. That was wrong for TUI apps —
xterm.js renders the byte stream serially on a fresh canvas, so replaying
historical bytes for an app like Claude Code paints the pre-TUI plain text
intro again, then the TUI repaints over it. The user perceives a duplicate.

Heuristic patches (trim to last screen-state-establishing escape, coalesce
into one frame, sync-output BEGIN detection, hide-cursor + clear-screen
prefix) closed some cases but never the structural one. The byte stream
just isn't enough information to reconstruct "current state" — only a
terminal emulator can do that.

The user's correct framing of the right architecture:

> Can we have a local client to keep listening on the session once the
> remote-control starts. Then everytime the web client call up a terminal
> we directly attach this local listener's result to the terminal.

That's tmux: a server-side terminal emulator holds the current grid state.
Clients attach and detach freely; on attach they receive the current state
directly (not byte history); live updates flow to all attached clients.

Concrete user scenarios this design must handle:

1. Quick navigation (Terminal A → Sessions → Terminal A): see the same
   state we had on the previous Terminal A view.
2. Tab close, browser killed by mobile OS, several minutes later, with
   shell commands typed on the laptop in between: see what's on the laptop
   pane RIGHT NOW.
3. TUI in progress (Claude Code thinking → reply): connect at any moment
   and see the current TUI frame.
4. Multiple phones connected to the same session: all see identical state.

Byte-replay can't honestly serve scenarios 2–4. Server-side terminal state
serves all four.

## 2. Architecture (one-paragraph summary)

The daemon runs a [`vt100::Parser`](https://crates.io/crates/vt100) per
session — a pure-Rust virtual terminal. Every byte that flows from the
laptop pane through `pty_reads_tx` → `pane_share` is fed to that parser
alongside being broadcast to live WS subscribers. On WS upgrade, the
handler asks the parser for its current screen state, serializes it to an
ANSI escape sequence, ships it to the new subscriber as the first Output
frame, then forwards live bytes from a fresh `broadcast::Receiver`. Both
"snapshot the state" and "subscribe to live" happen under the same map
mutex, so a chunk written concurrently with attach is delivered EITHER via
the snapshot OR via the live receiver — never both, never neither.

Resize: when the phone's xterm sends `sendControl({type:'resize',...})`,
the existing path forwards to the laptop pane; we additionally call
`parser.set_size()` on the daemon's emulator so the next snapshot reflects
the phone's viewport.

`https://crates.io/crates/vt100` — pure-Rust terminal emulator. Read its
API:

- `vt100::Parser::new(rows, cols, scrollback_lines)` — construct.
- `parser.process(&[u8])` — feed bytes (lock-free; parser is `&mut self`).

helper (~50 lines) that walks the visible grid, emits cursor moves + SGR
attributes + cells row-by-row, ends with the alt-screen toggle and final
cursor position.

## 4. Design choices to settle in §3 before coding

### 4.1 Where does the parser live, and how do bytes reach it?

Option A — daemon owns the parser, laptop calls a new API.
- `SessionEntry` gets `term: Arc<Mutex<vt100::Parser>>`.
- `pane_share` calls `registry.record_output(id, bytes)` (this method existed
  in the reverted Stage C and will need to be re-added with the SAME signature
  but doing different work — feed parser + broadcast).
- `subscribe` becomes `subscribe_with_state(id) -> (Bytes, broadcast::Receiver<Bytes>)`
  where the first element is the serialized snapshot.

Option B — keep `output_tx.send` direct; daemon taps the broadcast.
- `register_external` spawns a tap task that subscribes to its own
  `output_tx` and feeds the parser as bytes flow.
- Added complexity: tap-task lifecycle, ordering between tap-write and new
  subscriber's snapshot.

**Recommend A.** Single mutex, single source of truth, atomic snapshot.

### 4.2 Default size + resize handling

Initial `vt100::Parser::new(rows, cols, scrollback)` needs a size. Use
the `ExternalSessionSpec.initial_size` (currently a placeholder at 80×24).
On resize control frames from the phone, call `parser.set_size()` so the
next snapshot matches.

For phones with different viewport sizes connecting simultaneously: each
gets the same snapshot (sized to whatever the parser thinks). Acceptable
edge case; perfect-fit-per-client is out of scope.

### 4.3 Serializing the grid to ANSI

A small helper, ~50 lines:
1. If `screen.alternate_screen()`, emit `\x1b[?1049h`.
2. Emit `\x1b[2J\x1b[H` (clear + home).
3. For row in 0..rows:
   - Move cursor to row,0 (`\x1b[<row+1>;1H`).
   - For col in 0..cols, emit cell:
     - SGR sequence for fg/bg/attrs (only when changed).
     - The character.
   - End SGR (`\x1b[0m`) at row end if attrs were active.
4. Position cursor at `screen.cursor_position()` (`\x1b[<r>;<c>H`).
5. Emit cursor visible/hidden (`\x1b[?25h` or `\x1b[?25l`) per `screen.cursor_visible()`.

vt100 exposes everything needed via `Screen::cell(r,c) -> Option<&Cell>`
and `Cell` accessors (`fgcolor`, `bgcolor`, `bold`, `italic`, `underline`,
`reverse`, `contents` for the character).

### 4.4 Bounded scrollback on the parser

`vt100::Parser::new(rows, cols, scrollback_lines)` — pick a reasonable
`scrollback_lines` (1000? 5000?) so the user can scroll up in xterm to see
prior shell output. Memory cost: `cols × scrollback_lines × cell_size`
≈ 80 × 5000 × ~32 = 12 MiB per session. Maybe too big — start with 1000
(2.4 MiB) and tune.

Note: vt100's scrollback is internal to the parser. Our serializer only
emits the visible viewport. Phone's xterm has its own scrollback that
accumulates from live bytes. Phone scrollback ≠ daemon scrollback. The
daemon-side scrollback exists only to support multi-line cursor moves
within the visible grid; it does not get shipped on attach.

(If we later want "scroll up to see what happened during your absence",
that's a follow-up — emit a chunk of scrollback as part of the snapshot
along with the visible grid. Out of scope for this pass.)

## 5. Implementation phases

### Phase 1 — Add vt100 dependency, plumb the parser

1. `Cargo.toml` (workspace): `vt100 = "0.x"`.
2. `crates/omw-server/Cargo.toml`: depend on `vt100`.
3. `crates/omw-server/src/registry.rs`:
   - `SessionEntry` gains `term: Arc<Mutex<vt100::Parser>>`.
   - `register_external` constructs the parser using `spec.initial_size`.
   - `register` (owned-pty path) constructs similarly using `spec.cols`/`rows`.
   - New method `record_output(id, bytes: Bytes) -> Result<usize>`:
     - Take map lock.
     - Lock `term`, call `parser.process(&bytes)`.
     - Broadcast via `output_tx.send(bytes)`.
     - Return subscriber count.
   - Owned-pty's internal output pump in `register` switches from
     `output_tx.send` to calling `record_output` (or duplicates the
     parser-feed inline — judge based on borrow shape).
   - New method `subscribe_with_state(id) -> Option<(Bytes, broadcast::Receiver<Bytes>)>`:
     - Take map lock.
     - Lock `term`, build serialized snapshot.
     - Call `output_tx.subscribe()`.
     - Return both atomically.
   - New method `resize(id, rows, cols)` that calls `parser.set_size`.

4. `vendor/warp-stripped/app/src/omw/pane_share.rs`:
   - Change output pump from `output_tx_clone.send(bytes)` back to
     `registry_for_pump.record_output(session_id, bytes)`.
     (This was the Stage C change; it's correct here for a different
     reason — feed the parser.)

5. Unit tests in `omw-server`:
   - `record_output_feeds_parser`: register external, push `b"hello\n"`,
     check `screen.cell(0, 0..5)` reads "hello".
   - `subscribe_with_state_returns_serialized_grid`: register, push some
     output, attach, verify the returned bytes contain the expected
     cursor-positioning + cell content.
   - `subscribe_with_state_atomic_with_record_output`: tight loop hammering
     both — every byte ends up in EXACTLY ONE delivery (snapshot OR live
     receiver), never both, never neither.

### Phase 2 — Wire the WS handler

6. `crates/omw-remote/src/server.rs`: replace `subscribe(session_uuid)` with
   `subscribe_with_state(session_uuid)`. Pass the snapshot bytes through to
   `handle_authed_socket` alongside the receiver.

7. `crates/omw-remote/src/ws/pty.rs::handle_authed_socket`:
   - Accept `snapshot: Bytes` parameter.
   - In the upgrade closure, BEFORE spawning `pty_to_ws`:
     - If snapshot is non-empty, build one `Frame { kind: Output, payload: snapshot }`
       and `out_tx.send(Outbound::Frame(frame))`.
   - Spawn `pty_to_ws` as today.

8. Resize wiring: when an inbound `FrameKind::Control` carrying a resize
   payload arrives (already handled for the laptop pane), also call
   `inbound_registry.resize(session_id, rows, cols)`.

9. Integration test in `crates/omw-remote/tests/`:
   - `ws_attach_serializes_state`: register external session, push bytes
     producing a known screen state, open signed WS, verify the first
     Output frame's payload is a serialized grid that re-renders to the
     same state.
