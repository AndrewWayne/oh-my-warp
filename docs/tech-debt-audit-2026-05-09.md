## Architectural mental model

The omw delta is a local-first stack layered around a forked terminal client. Root Rust crates provide config, keychain access, CLI persistence, PTY/session registry, audit primitives, the loopback `omw-server`, and the tailnet-facing `omw-remote`. `apps/omw-agent` is the TypeScript pi-agent wrapper over stdio JSON-RPC; `apps/web-controller` is the browser controller served by `omw-remote`. The vendor fork embeds the root crates under `omw_local` and adds a narrow omw surface in `app/src/omw/**`, `app/src/ai_assistant/omw_*.rs`, and `app/src/settings_view/omw_agent_page*.rs`.

The fork posture is intentionally conservative: upstream `vendor/warp-stripped/` is inherited code and was not audited except for the explicit omw delta predicate. New omw behavior inside the fork should stay in omw-named files or carry the project's incremental-authorship header, because upstream sync is manual and broad upstream edits are expected to be overwritten/re-applied during restrip.

## Things that look bad but are actually fine

- `vendor/warp-stripped/app/src/ai_assistant/omw_agent_state.rs` uses file-level `#![allow(dead_code)]`; prior triage says this is Phase 3b/3c load-bearing, so it was not flagged.
- `omw-server::ExternalSessionSpec` and `SessionRegistry::register_external` are live via `vendor/warp-stripped/app/src/omw/pane_share.rs`; they are not dead API.
- `omw-remote::Capability::{AgentRead, AgentWrite, AuditRead}` are forward-compatible capability values and may already be minted; only the default `pty:write` grant is flagged here.
- `omw-pty/src/lib.rs` is large, but prior triage verified the writer, watcher, reader, and drop-safety threads are necessary.
- The `omw-keychain-helper` lib/main split is intentional because tests need to exercise the helper in process while the binary preserves the stdout-only secret invariant.
- `commands/agent.rs` and `commands/agent_runner.rs` split the REPL loop from per-turn spawning; merging them would blur two different runtime shapes.
- Many axum handlers are `async fn` without a local `await`; that is required by axum's `Handler` trait and was not treated as debt.
- `AgentCrashed`, `ApprovalDecision::Cancel`, and the `ExecCommand`/`CommandData`/`CommandExit` protocol variants are produced on live paths per the prior simplify audit.
- The `omw-cli` axum dependency is used by `commands/remote.rs`; it was not treated as an unused dependency even though `cargo machete` was unavailable.
- The ignored `crates/omw-server/tests/ws_concurrent.rs` test is documented as Beyond-v1 replay/broadcast semantics in the prior follow-up, so it stays out of the findings table.
- `serde_json::Value` in agent tool payload paths is acceptable at the dynamic pi-agent/tool boundary. The narrower TypeScript `any` in `warp-session-bash.ts` remains a finding because the method vocabulary is closed there.
- `unknown` at TS JSON trust boundaries is acceptable where immediately parsed by `parseProviderConfig` and `parsePolicy`.
- `docs/archive/**` and `RELEASE_NOTES_v*.md` contain historical brand and phase wording. The skill excludes those frozen artifacts, so they were not flagged.
- Upstream warnings from `vendor/warp-stripped` files outside the omw delta predicate, such as non-omw app modules, were ignored by design.
