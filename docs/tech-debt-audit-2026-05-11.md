## Architectural mental model

The omw delta is a local-first stack around a forked terminal client. Root Rust crates own config, keychain access, CLI persistence, PTY/session registry, audit primitives, the loopback `omw-server`, and the tailnet-facing `omw-remote`. `apps/omw-agent` is the TypeScript pi-agent kernel wrapper over stdio JSON-RPC, and `apps/web-controller` is the browser controller served by `omw-remote`.

The fork posture is deliberately narrow. Upstream `vendor/warp-stripped/` is inherited code and is outside this audit except for the omw delta predicate: `app/src/omw/**`, `app/src/ai_assistant/omw_*.rs`, `app/src/settings_view/omw_agent_page*.rs`, and any vendor file carrying omw incremental-authorship headers. New omw behavior should stay in those scoped files or carry the project header so a future upstream restrip does not bury local product logic in upstream noise.

## Things that look bad but are actually fine

- `vendor/warp-stripped/app/src/ai_assistant/omw_agent_state.rs` uses file-level `#![allow(dead_code)]`; prior triage says this is Phase 3b/3c load-bearing, so the allow itself was not flagged. The new finding is narrower: `start_with_config` missed the `AGENTS.md` prompt path.
- `omw-server::ExternalSessionSpec` and `SessionRegistry::register_external` are live via `vendor/warp-stripped/app/src/omw/pane_share.rs`; they are not dead API.
- `omw-remote::Capability::{AgentRead, AgentWrite, AuditRead}` are forward-compatible capability values and may already be minted; only the default `PtyWrite` grant is flagged.
- `omw-pty/src/lib.rs` is large, but prior triage verified the writer, watcher, reader, and drop-safety threads are necessary.
- The `omw-keychain-helper` lib/main split is intentional because tests need to exercise the helper in process while the binary preserves the stdout-only secret invariant.
- `commands/agent.rs` and `commands/agent_runner.rs` split the REPL loop from per-turn spawning; merging them would blur two runtime shapes.
- Many axum handlers are `async fn` without a local `await`; axum's `Handler` trait requires that shape.
- `AgentCrashed`, protocol `ApprovalDecision::Cancel`, and the `ExecCommand`/`CommandData`/`CommandExit` variants are produced on live paths per the prior simplify audit.
- The `omw-cli` axum dependency is used by `commands/remote.rs`; it was not treated as unused even though `cargo machete` was unavailable.
- The ignored `crates/omw-server/tests/ws_concurrent.rs` test is documented as Beyond-v1 replay/broadcast semantics in the prior follow-up, so it stays out of the findings table.
- `serde_json::Value` in agent tool payload paths is acceptable at the dynamic pi-agent/tool boundary. The narrower TypeScript `any` in `warp-session-bash.ts` remains a finding because the bash broker method vocabulary is closed.
- `unknown` at TS JSON trust boundaries is acceptable where immediately parsed by `parseProviderConfig` and `parsePolicy`.
- `docs/archive/**` and `RELEASE_NOTES_v*.md` contain historical brand and phase wording. The skill excludes those frozen artifacts.
- Upstream warnings from `vendor/warp-stripped` files outside the omw delta predicate, such as non-omw app modules, were ignored by design.
- The unused Tailscale `serve_https` and `unserve` helpers are not separate dead-code findings. They are kept as the documented alternative if the direct-tailnet-IP choice is reversed, and the real debt is the unresolved Serve-vs-direct-IP contract.
- `crates/omw-config/src/lib.rs` warning about oversize `AGENTS.md` uses `eprintln!`; that is currently a local config-file diagnostic, not a network/request-path observability issue.

- Should `/api/v1/audit/append` be mounted by the in-process server, feature-gated, or deleted? The prior simplify follow-up left this as a product decision.

- Is `omw-policy` meant to become the runtime policy engine, or should the TypeScript policy path remain canonical and the Rust crate be deleted?
- Is direct tailnet IPv4 now the official BYORC preview path, replacing Tailscale Serve, or should Serve be restored behind an opt-in?
- Should a newly paired phone be read-write by default for the preview demo, or should the BYORC read-only default invariant remain authoritative?
