# v0.4-thin: Functional Gaps and Fix Plans

Status: as of 2026-05-01, after wiring sub-phases 1–5 landed.
Owner: TBD (likely the same maintainer continuing into v0.4-cleanup).

This document enumerates the four functional gaps that prevent the v0.4-thin slice from being a "real" product experience, despite the underlying components passing 206 tests. Each gap includes symptoms, root cause, fix approach with concrete next steps, effort estimate, and acceptance criteria.

The gaps are listed in priority order — the order I'd close them in.

---

## Gap 1 — Phone gets a fresh shell, not the Warp terminal pane

### Symptom

User clicks the **Remote Control** button in `warp-oss`, pairs their phone, opens the Web Controller terminal. They start typing. But the shell they're driving is a **brand-new `cmd.exe` / `/bin/sh`** that `omw-remote` spawned via `omw-pty` — not the Warp terminal pane they were just looking at on their laptop.

This breaks the "remote control" promise. They're not *controlling* Warp; they're getting parallel access to a sibling shell.

### Root cause

The Phase E `WS PTY` handler and its Phase C-derived `omw-server::SessionRegistry` hold sessions that own their own `omw_pty::Pty`. Each `POST /api/v1/sessions` HTTP call (or each WS connect to an unknown id) spawns a fresh PTY. There's no path that says "register an external PTY-shaped thing that I, the Warp GUI, already own."

Warp's own PTY lives behind `vendor/warp-stripped/app/src/terminal/writeable_pty/pty_controller.rs::PtyController<T: EventLoopSender>`. It's a `PtyController` per terminal pane, owned by the pane's `TerminalModel`, fed via a `PtyWrite { Command, Bytes, AgentInput, TmuxCommand }` enum, with output going to the model's render pipeline.

To bridge: the omw-server registry needs to accept a session whose I/O is two channels (input writer + output broadcast) rather than a `Pty` value, and the Warp side needs to present its `PtyController` *as* that pair of channels.

### Fix approach

#### Step 1 — Extend `omw_server::SessionRegistry` with an external-source variant

```rust
// crates/omw-server/src/registry.rs

enum SessionSource {
    OwnPty(omw_pty::Pty),                                       // current
    External {
        input_tx: mpsc::Sender<Vec<u8>>,
        output_rx: broadcast::Receiver<Bytes>,
        kill: Box<dyn Fn() + Send + Sync>,
    },
}

impl SessionRegistry {
    pub async fn register_external(
        &self,
        name: String,
        spec: ExternalSessionSpec,
    ) -> Result<SessionId>;
}

pub struct ExternalSessionSpec {
    pub input_tx: mpsc::Sender<Vec<u8>>,
    pub output_rx: broadcast::Receiver<Bytes>,
    pub kill: Box<dyn Fn() + Send + Sync>,
    pub initial_size: PtySize,
}
```

`write_input` / `subscribe` route to the external channels for external sessions; `kill` calls the user-supplied closure. `resize` is a no-op for external sessions (or routes via a third channel — defer to Step 3).

#### Step 2 — Tests-first in `omw-server`

```rust
// crates/omw-server/tests/external_session.rs
//
// Build an mpsc + broadcast pair manually (no Pty involved).
// Register it as external. Verify:
//  - list/get see it as alive
//  - write_input pushes bytes to the input_tx
//  - bytes sent on the broadcast appear via subscribe()
//  - kill() closure is called on registry.kill(id)
```

This lets us validate the new code path without touching warp-stripped at all.

#### Step 3 — Hook into Warp's `PtyController`

Read `vendor/warp-stripped/app/src/terminal/writeable_pty/pty_controller.rs` end-to-end. Find:

- The output side: where `PtyController` reads from the master FD and pushes parsed bytes into the terminal model. Add a "fan-out" tap: clone every output chunk into a `broadcast::Sender<Bytes>` if a sharer is attached.
- The input side: `PtyController` consumes `PtyWrite { Bytes(...) }` from a channel. Wire an mpsc receiver from omw-server's `input_tx` directly into that channel.

A small new module `vendor/warp-stripped/app/src/omw/pane_share.rs`:

```rust
pub struct PaneShareHandle {
    pub session_id: SessionId,
    pub stop: Box<dyn Fn() + Send + Sync>,
}

pub fn share_pane(
    pane: &TerminalView,
    registry: &Arc<SessionRegistry>,
) -> Result<PaneShareHandle, ShareError> {
    let (input_tx, input_rx) = mpsc::channel::<Vec<u8>>(64);
    let (output_tx, _) = broadcast::channel::<Bytes>(256);

    pane.pty_controller().attach_input_channel(input_rx);
    let detach_output = pane.pty_controller().attach_output_tap(output_tx.clone());

    let registry_clone = registry.clone();
    let session_id = block_on(registry.register_external(
        pane.title().to_string(),
        ExternalSessionSpec {
            input_tx,
            output_rx: output_tx.subscribe(),
            kill: Box::new(move || {
                detach_output();
                let _ = block_on(registry_clone.kill(session_id));
            }),
            initial_size: pane.size(),
        },
    ))?;

    Ok(PaneShareHandle { session_id, stop: ... })
}
```

`attach_input_channel` and `attach_output_tap` are NEW public methods on `PtyController` — they don't exist today. Adding them is the load-bearing change inside warp-stripped.

#### Step 4 — UI surface

A new per-pane control: **"Share this pane"** in the terminal tab's right-click menu (or a small icon in the pane header). When clicked:

1. Calls `share_pane(...)`.
2. Stores the `PaneShareHandle` on the pane's metadata.
3. Shows a badge on the tab indicating it's shared (e.g., a small "🔗" icon — but per CLAUDE.md no emojis without explicit ask, so use a text/icon enum entry).
4. Click again → unshare (calls `handle.stop`).

The global "Remote Control" button (current Phase 5 wiring) keeps its meaning: starts/stops the daemon. Sharing individual panes is independent.

#### Step 5 — Web Controller surface

`apps/web-controller/src/pages/Sessions.tsx` (NEW) lists active sessions from `GET /api/v1/sessions`. Each row: name, alive, "shared from Warp pane: <pane-title>" badge for external sessions. Click → `/terminal/<hostId>/<sessionId>`.

This makes the phone-side experience: open Web Controller → see "main shell (shared from Warp pane: server.log tail)" → tap → terminal view shows the same bytes the laptop pane is showing in real time.

### Effort

- omw-server external-session refactor + tests: ~30 min.
- warp-stripped `PtyController` tap methods + safety read: ~1 hour.
- UI surface (per-pane share button + Sessions page): ~30 min.
- End-to-end: **~2-3 hours** in a focused session with overseer/executor TRD passes.

### Acceptance criteria

1. New test `crates/omw-server/tests/external_session.rs` passes.
2. `cargo build -p warp --bin warp-oss --features omw_local` still succeeds.
3. Manual smoke: open warp-oss, open a terminal pane, run `tail -f some.log`, click "Share this pane", phone connects, phone sees the live tail. Type on phone → laptop pane echoes the typed character.
4. Killing the warp pane terminates the shared session within 1 second on the phone.

## Gap 2 — Pair URL only in stderr, no QR popup

### Symptom

User clicks the Phone icon in warp-oss. The omw-remote daemon starts. The pair URL is `eprintln!`-ed to stderr of the launching console. If the user double-clicked `warp-oss.exe` from Explorer, there's no visible stderr at all — the URL is effectively lost.

Even when launched from PowerShell, the URL is mixed into a stream of unrelated GUI logs. The user has to scroll to find it, then manually type or copy it. This is unusable as a real UX.

### Root cause

The Phase 5 wiring scope was "first finish the wiring" — UI polish was explicitly deferred. The launcher just prints to stderr.

### Fix approach

A modal dialog opened by the Remote Control button. Two layouts depending on daemon state:

#### State: Stopped → click → starts daemon → Modal A

```
┌────────────────────────────────────────────┐
│  Remote Control                            │
├────────────────────────────────────────────┤
│  Status: Running on 127.0.0.1:8787         │
│                                            │
│  Pair URL:                                 │
│    http://127.0.0.1:8787/pair?t=...        │
│    [Copy]   [Show QR]                      │
│                                            │
│  Tailscale: ✓ laptop.tailnet.ts.net        │
│  (or "× tailscale not detected")           │
│                                            │
│  Paired devices: 0                         │
│                                            │
│  [Stop Daemon]   [Close]                   │
└────────────────────────────────────────────┘
```

#### State: Running → click → Modal A again (no toggle on click; explicit "Stop Daemon" button)

This avoids the "did clicking start it or stop it?" ambiguity of the current toggle.

#### Component pieces

1. **QR code rendering**: `qrcode = "0.14"` already a dep of `omw-cli`. Reuse. Render to PNG bytes via `qrcode::QrCode::new(url).render::<image::Luma<u8>>().build()` — or to SVG string for inline embedding in Warp's GUI.

2. **Modal infrastructure**: find an existing modal in warp-stripped (probably the settings dialog or command palette). Search: `grep -rn "Modal\|Dialog\|Popup" vendor/warp-stripped/app/src/ui_components/`. Adopt that pattern.

3. **Reactive paired-device count**: `OmwRemoteState` exposes a `tokio::sync::watch::Receiver<DaemonStatus>` (or whatever Warp's reactive Model trait is — see Gap 3 for the framework integration). The modal subscribes; when a redeem happens server-side, the count increments and the modal updates.

4. **Tailnet detection**: subprocess `tailscale status --json` parsed; surface the local hostname. If `tailscale` not found → show "Tailscale not detected — install from tailscale.com". (Implementation overlaps with Gap 4.)

5. **"Copy" button**: writes pair URL to system clipboard via Warp's clipboard helper.

#### Step-by-step

1. Add `qrcode` and `image` (or just `qrcode` with SVG output) to `vendor/warp-stripped/app/Cargo.toml [dependencies]`, gated behind `omw_local`.
2. Build the modal as `vendor/warp-stripped/app/src/omw/pair_modal.rs`. Read the settings dialog for the pattern.
3. Replace the current `ToggleOmwPair` toggle behavior:
   - First click on Phone icon → start daemon (if stopped) → open modal
   - Subsequent clicks while modal closed → just open modal
   - "Stop Daemon" button in modal → actual stop
4. The toast/eprintln stays as a fallback (and as a useful log for debugging) but is no longer the primary surface.

### Effort

- QR rendering + clipboard wiring: ~15 min.
- Modal layout + Warp UI integration: ~45 min (mostly reading the existing settings dialog for the pattern).
- Reactive paired-device count: ~15 min (after Gap 3's watch-channel landing).
- Tailnet detection display: ~15 min (after Gap 4's helpers).
- Total: **~1-2 hours**, blocked behind Gaps 3 and 4.

### Acceptance criteria

1. Click Phone icon on a running warp-oss → modal opens within 200 ms with the pair URL displayed.
2. "Show QR" reveals a QR code that scans correctly with a phone camera.
3. "Copy" places the URL on the clipboard.
4. Pairing a phone causes the "Paired devices" count to update from 0 → 1 within 1 s.
5. "Stop Daemon" terminates the daemon and updates the modal status.
6. No stderr URL prints required for normal use (kept only as a debug log).

## Gap 3 — Button label is static

### Symptom

The Remote Control button's tooltip is hardcoded to a single value. After the daemon starts, the tooltip still says "Toggle phone pairing" instead of "Stop phone pairing" (or similar). Same for the icon — there's no visual indicator that the daemon is running.

### Root cause

`OmwRemoteState` is a process-global `OnceLock<Arc<...>>` with `parking_lot::Mutex` inside. It's not integrated with Warp's reactive `Model` system — the UI never knows when the state changes, so it never re-renders the button.

### Fix approach

Two layers:

#### Layer 1 — Reactive state primitive

`OmwRemoteState` exposes a `tokio::sync::watch::Receiver<OmwRemoteStatus>` (or, if Warp uses `tokio::sync::broadcast`, that). Internal state mutations call `watch::Sender::send_replace(new_status)`. Subscribers see fresh values when they `.borrow()`.

Alternative: re-implement `OmwRemoteState` as a Warp `Model<T>` (or whatever the reactive entity trait is in Warp's UI framework — search for `impl Model for` or `Subscribable` or `Observable` in `vendor/warp-stripped/app/src/`). Then UI consumers get notified through Warp's native dispatch path.

The watch-channel option is portable and doesn't require deep Warp framework knowledge. The Model option is more idiomatic but takes longer to learn.

**Pick the watch-channel option for v0.4-cleanup-soft.** Polish to native Model in a later pass if needed.

#### Layer 2 — Button rebinds on state change

In `vendor/warp-stripped/app/src/ai/blocklist/agent_view/agent_input_footer/mod.rs` (where the button now lives), the button construction reads the current status:

```rust
let status = OmwRemoteState::shared().status();
let (icon, tooltip) = match status {
    OmwRemoteStatus::Stopped => (Icon::Phone, "Start phone pairing"),
    OmwRemoteStatus::Starting => (Icon::PhoneSpinner, "Starting..."),
    OmwRemoteStatus::Running { .. } => (Icon::PhoneActive, "Stop phone pairing"),
    OmwRemoteStatus::Failed { .. } => (Icon::PhoneError, "Pairing failed — click to retry"),
};
ActionButton::new().with_icon(icon).with_tooltip(tooltip).on_click(...)
```

Plus subscribe to the state changes via Warp's parent-rerender mechanism. The Phase 5 commit notes mention `notify_and_notify_children` was already invoked on the parent footer — this should already trigger a rerender; the missing piece is that the button construction reads stale state. Fix: read in the closure, not at construction time.

#### Step-by-step

1. Add `watch` channel to `OmwRemoteState`:
   ```rust
   pub struct OmwRemoteState {
       inner: Arc<RwLock<Inner>>,
       status_tx: tokio::sync::watch::Sender<OmwRemoteStatus>,
       pub status_rx: tokio::sync::watch::Receiver<OmwRemoteStatus>,
   }
   ```
2. Every state mutation publishes via `status_tx.send_replace(...)`.
3. UI button read state via `OmwRemoteState::shared().status_rx.borrow().clone()` at render time.
4. Tie a `Subscription` in the parent view to fire `notify_changed()` whenever status_rx fires. Warp's framework should re-render dependent widgets.
5. New icons (`PhoneActive`, `PhoneSpinner`, `PhoneError`) added to the icon enum if not already there. Or just vary the tooltip and rely on the modal for status visibility.

### Effort

- watch-channel plumbing: ~10 min.
- Reactive subscribe + render glue: ~20 min.
- Icon variants (or text-only state polish): ~10 min.
- Total: **~30-60 minutes**.

### Acceptance criteria

1. Stopped state: button tooltip says "Start phone pairing".
2. Starting state (transient, ~200 ms): tooltip "Starting...".
3. Running state: tooltip "Stop phone pairing", icon visually distinct (e.g., colored).
4. Click during Running → calls stop → tooltip transitions back to "Start phone pairing" within 500 ms.
5. Failed state: tooltip surfaces error briefly, button click retries.

## Gap 4 — Daemon listens on 127.0.0.1 only; no Tailscale Serve bootstrap

### Symptom

After clicking the Phone button, omw-remote listens on `127.0.0.1:8787`. The phone — even on the same tailnet — cannot reach it. The user has to:

1. Open a separate terminal.
2. Run `tailscale serve --bg --https=8787 http://127.0.0.1:8787` (and remember the syntax).
3. Look up the device's tailnet hostname via `tailscale status`.
4. Edit the pair URL: replace `127.0.0.1:8787` with `https://hostname.tailnet.ts.net`.

Many of those steps are unfamiliar; the failure mode (browser shows "connection refused") gives no diagnostic.

There's a security side too: the current `pinned_origin = "http://127.0.0.1:8787"` will reject WS upgrades whose `Origin` header is `https://hostname.tailnet.ts.net` — so even if the user manually does Tailscale Serve and updates the URL, the WS handshake will fail with `403 origin_mismatch`.

### Root cause

The Phase 5 launcher hardcodes `127.0.0.1:8787` as both the bind address and the pinned origin. There's no Tailscale integration, no detection, no reconfiguration when the network shape is different.

### Fix approach

Three sub-fixes:

#### 4a — Configurable bind address

Add a `bind_address` field to `OmwRemoteState`'s start config. Defaults to `127.0.0.1:8787`. The pair modal exposes a control to switch to `0.0.0.0:8787` (with a clear warning about LAN exposure) for users who don't have Tailscale.

#### 4b — Tailscale Serve bootstrap

New module `vendor/warp-stripped/app/src/omw/tailscale.rs`:

```rust
pub struct TailscaleStatus {
    pub installed: bool,
    pub running: bool,
    pub local_hostname: Option<String>,    // e.g., "laptop.tail-scale.ts.net"
    pub tailnet: Option<String>,           // e.g., "tail-scale.ts.net"
}

pub fn detect_status() -> TailscaleStatus {
    // 1. which("tailscale") — exists?
    // 2. tailscale status --json — running?
    // 3. parse local-hostname from JSON
}

pub fn serve_https(port: u16) -> Result<String, ServeError> {
    // shells out: tailscale serve --bg --https=8787 http://127.0.0.1:8787
    // returns the URL it served at
}

pub fn unserve(port: u16) -> Result<(), ServeError> {
    // tailscale serve --bg --https=8787 off
}
```

In `OmwRemoteState::start`:

1. Bind on 127.0.0.1:8787 as before.
2. Detect Tailscale via `detect_status()`.
3. If installed + running: invoke `serve_https(8787)`.
4. Construct the pair URL using the tailnet hostname (not 127.0.0.1).
5. Pin the origin to `https://<hostname>.<tailnet>.ts.net`.

In `OmwRemoteState::stop`: invoke `unserve(8787)` before tearing down the bind.

#### 4c — Multi-origin pinning

When both loopback access (for in-app debugging) and tailnet access (for the phone) are valid, the WS handshake's `Origin` check needs to accept either. Two options:

- **Option A**: Accept a `Vec<String>` of pinned origins, match if any pins.
- **Option B**: Match on a derived family — accept any origin where the host resolves to the local machine (loopback) OR the tailnet hostname.

Option A is simpler. Update `omw_remote::server::ServerConfig::pinned_origin` from `String` to `Vec<String>`, update the origin-check logic in `ws_handler` and the http-upgrade path. Update the existing Phase E origin tests and Phase 5 Tailscale-bootstrap test.

#### Step-by-step

1. **Add `omw-cli` flag**: `omw remote start --bind 0.0.0.0:8787 --tailnet-only` controls bind + Tailscale auto-serve. (Doesn't affect the warp-oss launcher — but useful for headless usage.)
2. **`tailscale.rs` module** with detection + serve/unserve helpers. Test the detection against a real `tailscale status --json` output (fixture-based).
3. **`OmwRemoteState::start` orchestration**: bind → detect → optionally serve → assemble pair URL → set pinned origins.
4. **`omw_remote::ServerConfig` API change**: `pinned_origin: String` → `pinned_origins: Vec<String>`. Migrate callers (Phase E and Phase 5 wiring). Update tests.
5. **Modal UI**: shows Tailscale status in the modal (per Gap 2). If detection says "not installed", show "Install Tailscale to enable phone access".

### Effort

- Tailscale CLI subprocess integration + fixture tests: ~1 hour.
- Pinned-origin Vec refactor + Phase E test updates: ~30 min.
- Modal display polish (after Gap 2): ~15 min.
- Total: **~1.5-2 hours**.

### Acceptance criteria

1. With Tailscale installed and running on the host, clicking Remote Control → modal shows pair URL with the tailnet hostname (`https://laptop.tailnet.ts.net/pair?t=...`).
2. Clicking phone-side that URL completes pairing and opens the terminal end-to-end.
3. With Tailscale not installed, the modal explicitly says so and falls back to 127.0.0.1 with an explanation.
4. Stopping the daemon also stops Tailscale Serve for that port (no leak).
5. Origin-check passes for both `https://laptop.tailnet.ts.net` (real phone) and `http://127.0.0.1:8787` (local dev/test).

## Cumulative effort estimate

| Gap | Effort (Claude-Code overnight pace) |
|---|---|
| 1 — Phone-attaches-to-Warp-pane | ~2-3 hours |
| 2 — Pair modal with QR | ~1-2 hours (after 3 + 4) |
| 3 — Reactive button | ~30-60 min |
| 4 — Tailscale bootstrap | ~1.5-2 hours |
| **Total** | **~6-8 hours = one overnight session** |

These run mostly in parallel; the only ordering constraint is **Gap 2 modal depends on Gap 4 for the "Tailscale status" line and on Gap 3 for the "Paired devices" reactive count**. Gap 1 is fully independent and could be the first thing landed (or run in parallel with the others).

For pacing reference: this whole session shipped 9 plan phases + 5 wiring sub-phases (~14 substantial components, 206+ tests). Each gap is smaller than a typical phase. Don't over-budget.

## Recommended sequence

1. **Gap 1 first** (most user-visible — without it, the slice is "phone gets *a* shell, not *the* shell"). Largest scope but unblocks the real promise.
2. **Gap 4 second** (without Tailscale auto-bootstrap, the demo requires four manual steps the user is unlikely to do).
3. **Gap 2 third** (modal depends on Gap 4's tailnet detection being available).
4. **Gap 3 last** (cosmetic; nice-to-have once the modal in Gap 2 is the primary surface anyway).

After all four close, the v0.4-thin slice becomes v0.4-cleanup minus the agent-integration items (`WarpSessionBashOperations` adapter is its own cluster and depends on v0.2's pi-agent — a separate effort, NOT in this gap doc).
