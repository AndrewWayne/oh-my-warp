This doc covers the next two features after the working pane-share demo
landed in commits `b17fb11` (auto-create on pair → land on Sessions),
`58f4272` (`share_self_pane`), `092c6ea` (pump tracing), `ec3377c`
(Stopped→Running gate), and `2a61ae5` (synchronized-output filter).

---

## 1. Goals

1. **Per-pane share semantics.** Each Warp pane has its own share state:
   shared or not shared. The Phone button on a pane reflects **that pane's**
   state — not the daemon's. Clicking shares or unshares **just that pane**.
   Currently every pane's button mirrors the global `OmwRemoteStatus`, so
   once any pane is shared, every other pane's button shows "Stop" and
   clicking it on another pane doesn't share that pane — instead it tries
   to global-toggle, which today re-pops a stale pair URL.
2. **Multi-pane share without re-pairing.** The phone pairs with the
   host **once**. From then on, the host can share multiple panes; the
   phone's `GET /api/v1/sessions` lists all of them. No new pair token
   per pane.
3. **Stop control.** Stopping a single share (from either side):
   - phone's terminal for that session disconnects with a clear status,
   - the underlying Warp pane keeps running on the laptop,
   - server resources are released (registry entry removed, pumps aborted),
   - the same pane can be re-shared later without restarting the daemon.
4. **Stop daemon stays distinct from unshare-this-pane.** Tearing down
   the entire omw-remote daemon (and revoking the pairing) is a
   different, less common action, surfaced via the QR/modal — not via
   accidentally clicking Phone on a random pane.

## 2. Current state survey

### 2.1 Share trigger (and the actual UX bug)

`vendor/warp-stripped/app/src/terminal/view/use_agent_footer/mod.rs:280-345`
(the `ToggleOmwPair` arm in `handle_use_agent_footer_event`):

- Click Phone → if daemon `Stopped|Failed` → `state.start()`.
- **Only on a fresh `Stopped → Running` transition** (commit `ec3377c`):
  `ctx.spawn(Timer::after(0), |me, _, ctx| share_self_pane(me, ctx, registry, runtime))`
  registers the **active** pane (whose button was clicked) and stores
  its `PaneShareHandle`.
- After that — on **every** subsequent click from **any** pane —
  the handler still runs the post-start surface code:
  - copies the pair URL to clipboard (line ~342),
  - opens `pair_browser::open_pair_page` to a local HTML page with the
    QR (line ~349).

**The bug the user is hitting.** The Phone button itself reads
`OmwRemoteState::shared().status()` — a process-global value. So:

1. User clicks Phone on pane A → daemon starts → `OmwRemoteStatus::Running` →
   `share_self_pane(A)` registers pane A.
2. Pane B's button (reading the same global status) now shows
   "Stop phone pairing" — even though pane B isn't shared.
3. User clicks Phone on pane B expecting to share pane B. The handler
   sees the daemon is already Running, falls through past the
   `state.start()` and `share_self_pane` blocks (gated to
   Stopped→Running), and re-pops the pair URL/QR via clipboard +
   `open_pair_page`. The URL is the **stale, already-used pair
   token** — phone scanning it gets `token_already_used`. Pane B is
   never shared.

So the real symptoms are:
- "Stop" button label is **misleading** on every non-shared pane.
- Clicking it pops a useless QR (stale token) and does **not** share
  that pane.
- There's no path from one shared pane to a second shared pane
  without restarting the daemon.

Reactive button label (`use_agent_footer/mod.rs` near line 360):
the button subscribes to `state.subscribe_status_stream()` (per
Gap 3 / commit `5df3476`) which is keyed only on global daemon
status — so flipping behaviour can only follow that, not per-pane
share state.

### 2.2 Share lifecycle

`vendor/warp-stripped/app/src/omw/pane_share.rs::share_pane`:

- Builds an `mpsc::Sender<Vec<u8>>` (input) + `broadcast::Sender<Bytes>`
  (output) pair, spawns input/output pumps wired to Warp's
  `event_loop_tx` + `pty_reads_tx`.
- Calls `registry.register_external(spec)` with a kill closure that
  aborts the pump JoinHandles.
- Returns `PaneShareHandle { session_id, stop }` where `stop` is a
  closure that asynchronously calls `registry.kill(session_id)`.

`vendor/warp-stripped/app/src/omw/remote_state.rs`:

- `Inner.pane_shares: Vec<PaneShareHandle>` (line 94 in current code,
  one slot per share).
- `OmwRemoteState::store_pane_shares(handles)` appends.
- `OmwRemoteState::stop()` calls `pane_shares.clear()` *before*
  dropping `pty_registry` (commit `1272ce6`'s comment), so each
  handle's stored kill closure can call `registry.kill(id)` against
  a still-live registry.

**Important caveat:** today `pane_shares.clear()` just drops the
`PaneShareHandle`s. Their `stop` closures are NOT fired automatically
on drop — `Box<dyn Fn() + Send + Sync>` has no `Drop` semantics that
runs the closure. The pumps still get aborted, but **only via the
Drop chain**: registry HashMap drops → SessionEntry drops →
External `kill: Option<Box<dyn Fn>>` drops without firing → captured
`pumps_for_kill` Arc drops → JoinHandles drop → tasks cancelled.
This works in practice (tasks are cancelled on JoinHandle drop) but
is implicit. See §4.4 for the recommended fix.

### 2.3 Stop pathway

**Daemon side (`crates/omw-server/src/registry.rs::SessionRegistry::kill`,
lines 388-422):**

1. Removes the entry from the HashMap atomically.
2. For `External` source: calls the `kill` closure exactly once
   (`kill: Option<Box<dyn Fn>>`).
3. The closure aborts both pumps via the captured `pumps_for_kill`
   handles.
4. Marks `entry.alive = false` so racing GETs see it as dead.

**HTTP route (`crates/omw-remote/src/http/sessions.rs::delete`,
lines 140-170):** signed DELETE on `/api/v1/sessions/:id` →
`registry.kill(uuid)` → returns 204 on success, 404 on unknown id.
Requires `Capability::PtyWrite` (default token has it per
`http/pair_redeem.rs:65`).

**Web Controller side (`apps/web-controller/src/pages/Sessions.tsx`):**
the Stop button calls `deleteSession(pairing, id)` (defined in
`lib/sessions.ts`). 404 is silently treated as "already gone." The
list refreshes after.

**What works today:** clicking Stop on the Sessions page DOES kill
the daemon-side share. The output broadcast closes; if the phone is
currently on the terminal page for that session, its WS receives
the close and Terminal.tsx surfaces "disconnected" status with a
Retry button. The underlying Warp pane is unaffected. ✅

**What's broken or unclear:**
1. After Stop, `OmwRemoteState.pane_shares` still holds the dead
   `PaneShareHandle`. Small memory leak; cleared on daemon stop.
2. There's no host-side "stop sharing this pane" affordance.
3. There's no visual indication on the laptop that a pane is being
   shared.
4. The Sessions.tsx auto-navigate (when exactly 1 alive session
   exists) might bounce the user back into a re-shared session
   unintentionally if reshare-on-click ever triggers. Need to
   guard.

### 2.4 Web Controller surfaces

- `pages/Home.tsx` — list paired hosts → click "Open" navigates to
  `/host/:hostId`.
- `pages/Sessions.tsx` — lists sessions for one host (5 s auto-
  refresh). Per-session: "Open" → `/terminal/:hostId/:sessionId`,
  "Stop" → DELETE. Auto-navigate fires once on initial load when
  exactly one alive session is present.
- `pages/Terminal.tsx` — xterm.js + signed WS bridge. Disconnect
  shows Retry.

## 3. Feature 1: Multi-pane share

### 3.1 Design

Adopt the user's "Option A" — keep the existing host-level pairing
(one phone, one host, one pair flow), but make **each pane's Phone
button operate on that pane's individual share state**. The phone is
already paired; multiple panes register as separate sessions in the
registry and the phone's Sessions page lists them all.

**Per-pane button state matrix** (replaces the current global toggle):

| Daemon status | This pane shared? | Button shows | Click action |
|---|---|---|---|
| `Stopped` / `Failed` | (n/a) | "Share with phone" | Start daemon + share THIS pane + surface QR/URL once for first-pair |
| `Starting` | (n/a) | "Starting…" (disabled) | — |
| `Running` | **No** | "Share with phone" | Share THIS pane. **Do NOT re-pop the QR.** |
| `Running` | **Yes** | "Stop sharing" | Unshare THIS pane only. Daemon stays up if any other pane is still shared; otherwise daemon stops. |

The QR/clipboard write only fires on the daemon-start click (case 1
above). Subsequent shares are silent — the phone is already paired
and will pick the new session up via the Sessions page (auto-refresh
every 5 s, or manual refresh).

### 3.2 What used to be the "global stop"

The current Phone button conflates two operations: (a) toggle pane
sharing, (b) toggle the daemon. After this redesign, (a) is the
button's only job. The daemon shutdown move to:
- the QR/pair-modal "Stop daemon" button (already proposed by the
  Gap-2 deferred dialog work — `pair_modal.rs` documents it), or
- a separate omw-cli command `omw remote stop` (already exists).

If no `pair_modal.rs`-backed dialog ships in this iteration, the
daemon auto-stops when the **last** shared pane is unshared (case
"Running, this pane shared" in the table → unshare → if pane_shares
becomes empty, also stop the daemon). This keeps the UX simple:
no shared panes = no daemon = no leakage.

### 3.3 Identity & idempotency

We need to recognise "this pane is already shared" so a second click
doesn't double-register.

**Use `TerminalView::view_id()` (an `EntityId`)** as the share key.
There is already a `pub(crate) fn view_id(&self) -> EntityId` accessor
at `vendor/warp-stripped/app/src/terminal/view.rs:23668`.

Change `OmwRemoteState`:

```rust
// remote_state.rs
struct Inner {
    ...
    // Was:  pane_shares: Vec<PaneShareHandle>
    // Now:  keyed by the TerminalView's EntityId so a second Phone-
    //       click from the same pane is a no-op (idempotent).
    pane_shares: std::collections::HashMap<EntityId, PaneShareHandle>,
    ...
}
```

API surface:

```rust
impl OmwRemoteState {
    /// Insert one share keyed by the source TerminalView's id. If a
    /// share already exists for that id, returns false and the new
    /// handle is dropped (the caller may want to keep the old one;
    /// share_self_pane is responsible for not bothering to call this
    /// when already-shared).
    pub fn store_pane_share(&self, view_id: EntityId, handle: PaneShareHandle) -> bool;

    /// True iff a pane with this view id is already in the share map.
    pub fn is_pane_shared(&self, view_id: EntityId) -> bool;

    /// Remove and explicitly fire the stop closure for this pane. Used
    /// by §3.4 host-side unshare and by the §4.3 cleanup-after-Stop
    /// reconciliation.
    pub fn unshare_pane(&self, view_id: EntityId);
}
```

Drop `pane_shares: Vec<...>` and the `store_pane_shares(Vec<...>)`
multi-handle setter we have today.

### 3.4 Trigger change — per-pane dispatch

`use_agent_footer/mod.rs::ToggleOmwPair` is reshaped from "start daemon
+ pop QR" into a four-arm dispatcher:

```rust
let view_id = self.view_id();
let was_stopped = matches!(state.status(), Stopped | Failed { .. });
let was_shared = state.is_pane_shared(view_id);

match (was_stopped, was_shared) {
    // Daemon is up AND this pane is already shared -> unshare it.
    (false, true) => {
        state.unshare_pane(view_id);
        // If that emptied the share map, also stop the daemon (or
        // leave it running until the user explicitly stops via the
        // QR modal — see §3.2).
        if state.share_count() == 0 {
            let _ = state.stop();
        }
        // Toast for clarity. No QR pop.
        log::info!("omw: unshared pane {view_id}");
    }

    // Daemon is up but this pane isn't shared yet -> share it.
    (false, false) => {
        // Defer via Timer::after(0) — same reentry safety reasoning
        // as the original share_self_pane wiring.
        ctx.spawn(Timer::after(Duration::ZERO), move |me, _, ctx| {
            if let (Some(registry), Some(runtime)) =
                (state.pty_registry(), state.runtime_handle())
            {
                if let Some(handle) = share_self_pane(me, ctx, registry, runtime) {
                    state.store_pane_share(me.view_id(), handle);
                }
            }
        });
        // No QR pop — phone is already paired.
    }

    // Daemon is down -> start it AND share this pane. ONLY this case
    // surfaces the QR/clipboard, because it's the actual pair moment.
    (true, _) => {
        if let Err(e) = state.start() {
            log::warn!("omw: start failed: {e}");
            return;
        }
        ctx.spawn(Timer::after(Duration::ZERO), move |me, _, ctx| {
            if let (Some(registry), Some(runtime)) =
                (state.pty_registry(), state.runtime_handle())
            {
                if let Some(handle) = share_self_pane(me, ctx, registry, runtime) {
                    state.store_pane_share(me.view_id(), handle);
                }
            }
        });
        // First-pair surface: QR + clipboard + open browser. The
        // existing pair_browser path stays here (this is the only
        // remaining call site).
        surface_pair_modal(state, ctx);
    }
}
```

The `surface_pair_modal` helper extracts the existing clipboard /
`pair_browser::open_pair_page` / eprintln-toast block from the
current handler. It only runs in the `(true, _)` arm.

### 3.5 Button label binding (the "every pane shows Stop" fix)

Today the button reads from `state.subscribe_status_stream()`, which
only carries `OmwRemoteStatus`. We need a per-pane state too.

**Recommendation: compute the button label at render time** from
*both* the daemon status and `state.is_pane_shared(view_id)`, and
re-render whenever either changes. Two new wires:

1. **Carry `view_id` into the button construction.** The agent
   footer already has access to `self.terminal_view_id` (an
   `EntityId`) — pass it through to the `ActionButton` builder for
   the omw button.

2. **Add a per-pane share-state observation.** Two options:

   - **Option α: piggyback on the existing status_stream** — make
     `OmwRemoteState` emit a richer event that includes "the share
     map changed" so the same `subscribe_status_stream` consumer
     can re-render. Risks: changes the `OmwRemoteStatus` enum
     (consumed by the modal too), bigger blast radius.

   - **Option β: a separate watch channel on `OmwRemoteState` for
     share-map changes.** A `tokio::sync::watch::Sender<u64>`
     whose value is a "share-map version" counter, bumped whenever
     `store_pane_share` or `unshare_pane` runs. The button
     subscribes via a sibling `subscribe_share_stream()` and
     re-renders on tick. Cheap, doesn't disturb existing wiring.

   **Pick Option β.** ~30 lines on `OmwRemoteState` + ~10 lines in
   the button construction.

Pseudocode for the button:

```rust
let initial_status = state.status();
let initial_shared = state.is_pane_shared(view_id);
// Two streams: status changes + share-map changes. Both fire a
// re-render closure that recomputes (icon, tooltip).
let status_rx = state.subscribe_status_stream();
let share_rx = state.subscribe_share_stream();
let view_id_for_render = view_id;
ctx.spawn_stream_local(
    select_stream(status_rx, share_rx),
    move |me, _, ctx| {
        let st = OmwRemoteState::shared().status();
        let shared = OmwRemoteState::shared().is_pane_shared(view_id_for_render);
        let (icon, tooltip) = match (st, shared) {
            (Stopped | Failed { .. }, _)        => (Icon::Phone, "Share with phone"),
            (Starting, _)                       => (Icon::PhoneSpinner, "Starting…"),
            (Running { .. }, false)             => (Icon::Phone, "Share with phone"),
            (Running { .. }, true)              => (Icon::PhoneActive, "Stop sharing"),
        };
        // Apply icon + tooltip + active-state to the ActionButton.
        ...
    },
);
```

### 3.6 Daemon-stop affordance

After §3.3, the Phone button no longer stops the daemon. We need
*some* way to stop it (otherwise it leaks until warp-oss exits).
Two options for v0.5-thin scope:

- **Auto-stop when share map empties.** Already in §3.3's `(false,
  true)` arm: after `unshare_pane`, if `share_count() == 0`, call
  `state.stop()`. Simple; matches "no shares = no daemon."
- **Explicit "Stop daemon" button in the pair modal.** The existing
  `pair_modal.rs` text formatter already documents a "Stop daemon"
  affordance; the actual `View<>`-backed dialog is deferred. Land
  whenever the dialog work resumes.

Both can coexist: auto-stop is the safe default, modal-Stop is the
explicit override during pairing.

### 3.7 Web Controller side

No change required for the basic multi-share story:

- `Sessions.tsx` already lists every session it sees from
  `GET /api/v1/sessions`. With multi-pane share, the list will
  contain all of them.
- The auto-navigate-when-1-alive-session is intentionally specific to
  the single-pane case; with N>1 panes shared, the user gets the list
  and picks one.

**Refinement to consider:** since each external session's `name`
field defaults to `"active-pane"` (commit `58f4272`'s
`spawn_share_and_collect`), the list shows duplicate names. Pass a
better name through:

- Easy: include the view's pwd or pane index. Use
  `TerminalView::pwd_if_local(ctx)` (defined at view.rs:21336) when
  available, fall back to `format!("pane-{}", view_id)`.
- Better: include the active block's command. Available via the
  block_list model; a small accessor.

### 3.8 Multi-window note

`Workspace` is per-window; so is the Phone button (it lives in each
TerminalView's footer). Multi-pane share within one window via
Option B works without any cross-window plumbing because each click
operates on the active pane in its own window. Cross-window
"see all panes from every window" is a separate concern, deferred.

## 4. Feature 2: Stop-control reliability

### 4.1 Phone-side Stop today

Sessions.tsx Stop button → DELETE `/api/v1/sessions/:id` →
`registry.kill(uuid)` → fires the External `kill` closure → aborts
the pumps → SessionEntry removed from registry. WS subscribers see
`output_rx.recv()` return `Closed` → `pty_to_ws` task breaks → writer
closes the WS sink → phone's Terminal.tsx fires `onClose` → status
goes "disconnected".

**This works.** The user's "Stop button should take effect" remark
suggests one of:

(a) **The Sessions page's Stop button is being clicked, but the
    list still shows the entry** (refresh race).
    Cause: `deleteSession` resolves on 204, then `refresh(pairing)`
    runs. If the registry has propagated the removal, the new list
    omits the entry. If a refresh hadn't yet shown it dead, this
    looks fine. Race window is tiny.

(b) **The Terminal page's WS doesn't surface a clean "session
    stopped" message** — it just goes "disconnected (1006)" and
    offers Retry. The user retries, gets `session_not_found`,
    confused.
    See §4.3 for the proposed fix.

(c) **No host-side affordance to stop sharing a pane.** The user
    expected a Stop button in warp-oss too. Defer (§3.4).

### 4.2 OmwRemoteState reconciliation

When the phone calls DELETE `/api/v1/sessions/:id`, the daemon's
registry kills the session. **But `OmwRemoteState.pane_shares`
still holds the now-orphan PaneShareHandle.** No correctness bug
(the next call to that handle's stop is a NotFound no-op), but it's
untidy and can confuse the §3.2 idempotency check (
`is_pane_shared(view_id)` would say "yes" even though the daemon
side is already torn down).

**Fix:** wire a notification from the registry back to
`OmwRemoteState` whenever a session is killed. Two design options:

| Option | Approach | Pros | Cons |
|---|---|---|---|
| **R1. Active poll** | After every Phone-click (or via a tokio interval), reconcile `pane_shares` against `registry.list()` and remove handles whose session_id is gone. | No new infrastructure. | Latency / staleness; idempotency check is briefly wrong. |
| **R2. Callback hook** | Add `register_external` accept a "kill callback" that runs *after* the share teardown completes. The callback removes the handle from `pane_shares` keyed by view_id. | Immediate cleanup. | Touches `omw_server::ExternalSessionSpec` (cross-crate change). |
| **R3. Drop on PaneShareHandle** | Add `impl Drop for PaneShareHandle { fn drop(&mut self) { (self.stop)() } }`. Then dropping = unsharing. `OmwRemoteState.unshare_pane(view_id)` removes-and-drops. | No new types; idempotency is automatic. | If we want the registry to be source of truth, we still need to *trigger* the drop when the registry kills externally (e.g., DELETE from phone). |

**Recommendation: R3 (Drop on PaneShareHandle) PLUS a periodic
reconcile.** The Drop semantics make `unshare_pane` clean and
foolproof; the periodic reconcile catches the case where the phone
killed the session via DELETE without telling us. R2 is cleaner but
crosses crate boundaries; defer it as a v1 hardening.

### 4.3 Web Controller side: graceful "session stopped"

When a session is killed remotely (e.g., another client triggered
DELETE, or the host called `unshare_pane`), the phone's Terminal.tsx
sees the WS close with code 1006 (abnormal). The user gets a generic
"disconnected" alert and a Retry button that 404s.

**Fix sketch in Terminal.tsx:**

```tsx
connection.onClose((info) => {
  if (cancelled) return;
  if (info.code === 1006 || info.code === 1011) {
    // Session is likely gone. Try a refresh of the host's session
    // list; if our sessionId is no longer present, navigate back to
    // /host/:hostId and tell the user the session ended.
    void listSessions(pairing).then((sessions) => {
      if (!sessions.some(s => s.id === sessionId && s.alive)) {
        navigate(`/host/${encodeURIComponent(hostId!)}`, { replace: true });
        return;
      }
      // Session is alive but our WS dropped — show retry.
      setErrorMsg(`Connection closed (${info.code}). Tap Retry to reconnect.`);
      setStatus("disconnected");
    });
  } else {
    setErrorMsg(`Connection closed (${info.code})`);
    setStatus("disconnected");
  }
});
```

This auto-navigates back to the Sessions page when the underlying
session is gone, instead of stranding the user on a dead Terminal
view.

### 4.4 Cleanup: explicit kill on stop()

Today `OmwRemoteState::stop()` calls `pane_shares.clear()` and
relies on Drop chains to abort tasks. With the §3.2 HashMap and the
§4.2 R3 Drop impl, this becomes explicit:

```rust
// remote_state.rs
pub fn stop(&self) -> Result<(), String> {
    let task = {
        let mut g = self.inner.lock();
        self.set_status(&mut g, OmwRemoteStatus::Stopped);
        // PaneShareHandle::Drop fires its `stop` closure, which calls
        // registry.kill(id). We do this BEFORE dropping pty_registry
        // so each kill races against a still-live registry.
        g.pane_shares.clear();
        g.pty_registry = None;
        g.serve_task.take()
    };
    if let Some(task) = task { task.abort(); }
    Ok(())
}
```

## 5. Implementation plan

### Stage A — keying the share map by view id (foundation for both features)

1. **`vendor/warp-stripped/app/src/omw/remote_state.rs`**
   - Replace `pane_shares: Vec<PaneShareHandle>` with
     `HashMap<EntityId, PaneShareHandle>`.
   - Drop `store_pane_shares(Vec<...>)`. Add
     `store_pane_share(EntityId, PaneShareHandle) -> bool`,
     `is_pane_shared(EntityId) -> bool`,
     `unshare_pane(EntityId)`.
   - Update `stop()` to call `g.pane_shares.clear()` (HashMap clear
     same semantic).

2. **`vendor/warp-stripped/app/src/omw/pane_share.rs`**
   - Add `impl Drop for PaneShareHandle` that fires the stop closure
     once. Use a `Cell<Option<Box<dyn FnOnce>>>` or similar so it's
     fired exactly once. Detail: the closure today is `Fn` so it can
     be called multiple times; switch to `FnOnce` and consume on
     drop.

3. **Tests** (add to `pane_share.rs::tests`)
   - Drop calls registry.kill exactly once.
   - HashMap insertion is idempotent: second call with same
     EntityId is a no-op.

### Stage B — per-pane click dispatch + button label

4. **`vendor/warp-stripped/app/src/omw/remote_state.rs`**
   - Add `share_count(&self) -> usize` (returns
     `self.inner.lock().pane_shares.len()`).
   - Add a sibling watch channel `share_tx:
     watch::Sender<u64>` whose value is bumped every time
     `store_pane_share` or `unshare_pane` mutates the map. Add
     `subscribe_share_stream(&self) -> ...` mirroring
     `subscribe_status_stream`.

5. **`vendor/warp-stripped/app/src/terminal/view/use_agent_footer/mod.rs`**
   - Restructure the `ToggleOmwPair` arm into the four-arm dispatcher
     in §3.4: `(was_stopped, was_shared)` match, with `share_self_pane`
     in two arms (`(false, false)` and `(true, _)`), `unshare_pane` in
     `(false, true)`, and the QR/clipboard/`open_pair_page` block

## 7. Open questions / non-goals

- **Per-message ordering across multiple shares.** Out of scope —
  each session's WS is independent. The phone is responsible for
  sequencing within a single session.
- **Reactive new-pane share.** Deferred to Option D in a follow-up.
- **Cross-window sharing.** Each Phone button shares within its own
  window. Multi-window iteration left for v0.5+.
- **Authorization granularity.** All paired devices currently see
  every share. Per-pane ACLs are a v1.0 hardening item.
- **Resize negotiation.** `omw_pty::PtySize` is a placeholder in
  ExternalSessionSpec today. The phone's xterm.js dimensions don't
  propagate back to Warp's pane (Warp dimensions stay set by warp-
  oss). For TUI apps this can cause line-wrap mismatches; track as
  a known limitation, not a blocker.
